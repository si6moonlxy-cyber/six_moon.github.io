## 八、迭代原则

> 项目执行时需遵守的原则。

1. **成本-收益平衡**：低成本高用户倾向功能优先
2. **版本周期控制**：单版本周期约 1-2 周，庞大功能拆分
3. **直观收益导向**：每次迭代需产生可衡量的直观收益
4. **项目三角法则**：始终平衡成本、时间、范围、质量

## 附录：PRD 文档生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                      PRD 生命周期流程                          │
└─────────────────────────────────────────────────────────────┘

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │   创建    │───▶│   评审    │───▶│   修改    │───▶│   确认    │───▶│   归档    │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
       │               │               │               │               │
       ▼               ▼               ▼               ▼               ▼
  产品经理撰写    评审会（PM/   根据评审意见    评审通过后    归档至需求
  初始需求文档    设计/研发/   修改文档        确认发布      知识库
                  测试）                         ⭐
                                                 │
                    ┌───────────────────────────┘
                    ▼
               ┌──────────┐    ┌──────────┐
               │  变更    │───▶│  重新评审  │
               └──────────┘    └──────────┘
```

### 流程说明

| 阶段 | 负责人 | 输入 | 输出 | 节点标准 |
| --- | --- | --- | --- | --- |
| **创建** | 产品经理 | 需求来源（用户反馈/数据分析/竞品分析/业务规划） | PRD 草稿 v0.1 | 需求背景清晰、目标明确 |
| **评审** | PM + 设计 + 研发 + 测试 | PRD 草稿 | 评审意见 + 修改建议 | 功能可行、技术成本合理、体验设计通过 |
| **修改** | 产品经理 | 评审意见 | PRD 修订版 v0.2+ | 解决评审中发现的问题 |
| **确认** | 所有评审人 | 修订版 PRD | 确认签字/文档状态更新为「已确认」 | 全员无异议 |
| **归档** | 产品经理 | 已确认 PRD | 归档文档 + 需求关联 | 文档存入知识库，需求进入开发 |

### 变更流程

> 已发布 PRD 如需变更，需走以下流程：

```
变更申请 → 评估影响 → 重新评审 → 确认变更 → 更新版本号
```

| 变更类型 | 处理方式 |
| --- | --- |
| 小幅修正（错别字/格式） | 直接修改，版本号不变 |
| 功能增删（不影响核心流程） | 修改后版本号 +0.1 |
| 核心功能变更/范围调整 | 重新走完整评审流程，版本号 +1.0 |
